## 2026-02-19T10:00:00Z

Starting implementation of priority-aware downstream scoring for the `next` command.

**Problem:** Currently, `ScoreTask()` applies the downstream bonus (+15 max) and critical-path bonus (+15) regardless of the priority of the downstream tasks. A low-priority task with 5 low-priority downstream tasks scores 40 (10 + 15 + 15), beating a standalone medium-priority task at 20.

**Approach:**
- Compute the max priority in the downstream chain for each task
- Scale downstream and critical-path bonuses by a multiplier based on that max priority:
  - critical/high downstream → full bonus (1.0)
  - medium downstream → 0.5 bonus
  - low downstream → 0.25 bonus
- This requires passing the task graph (or task map + downstream info) into `ScoreTask()`
- The `computeDownstreamCounts` function already builds a graph; extend it to also capture max downstream priority

**Plan:**
1. Add `computeDownstreamInfo()` that returns both counts and max priority per task
2. Modify `ScoreTask()` signature to accept max downstream priority
3. Scale downstream and critical-path bonuses accordingly
4. Add tests for the scenarios described in the task
