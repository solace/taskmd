## 2026-02-16T10:00:00Z

Started implementation of fix for list command table column alignment.

**Problem:** `outputTable()` uses `strings.Repeat("-", 10)` for all separator columns, regardless of actual header/content width. `tabwriter` pads shorter entries with spaces, so separators appear too short for wide columns like "title".

**Approach:** Compute the max width per column by scanning both headers and data values, then generate separator dashes to match. Need to handle ANSI color codes by stripping them when measuring width, since colored values appear wider in raw string length but render at their visible width.

## 2026-02-16T11:00:00Z

Completed implementation.

**Changes:**
- `apps/cli/internal/cli/list.go`: Updated `outputTable()` to compute max visible width per column across headers and plain-text data values, then generate separator dashes of that width instead of hardcoded 10
- `apps/cli/internal/cli/list_test.go`: Added `TestListCommand_SeparatorAlignment` test validating that separator dashes are at least as wide as headers and longest data values

**Decisions:**
- Used `getColumnValue()` (plain text) rather than `colorizeColumn()` (ANSI-wrapped) for width calculation, so separators match the visible width correctly
- ANSI color codes with tabwriter is a pre-existing concern (tabwriter counts ANSI bytes as column width), but with no-color in piped/non-TTY contexts the output is perfectly aligned, and in TTY mode the slight extra padding from ANSI codes is cosmetically acceptable
